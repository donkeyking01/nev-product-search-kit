# 02 Collect Evidence

输出 `02_evidence_collection.md`。围绕 `01_research_scope.md` 联网收集证据，不要超范围发散。

必要时先读 `../references/evidence-rules.md`。

## 这一阶段的原则

- 这是强制联网阶段。不要只凭既有知识或模糊记忆写证据。
- 先收集事实，再做归纳；不要在这里提前写产品结论。
- 证据要覆盖市场、产品、用户声音、竞品四个维度。
- 关键事实必须记录来源、日期和可追溯线索。

## 必须联网的情况

以下情况默认必须联网，而不是直接依赖模型记忆：

- 市场规模、销量、份额、价格、配置、发布时间等会变化的信息
- 当前在售车型、竞品组合、版本迭代、权益政策
- 用户近期口碑、论坛讨论、社媒反馈
- 官方公告、财报、媒体评测、研究报告
- 用户明确要求“查一下”“搜一下”“验证一下”“看最新”

如果信息存在明显时效性，就把联网检索当作默认动作。

## Skill 和工具调用顺序

优先使用当前环境里已经可用、且最贴近任务的 skill / 工具。

### 1. 联网搜索

- 如果当前环境提供通用 Web 搜索或浏览能力，优先使用它。
- 先用搜索工具发现来源，再打开具体页面读取细节。
- 不要只看搜索摘要；关键事实要进入原文页面核对。

### 2. 浏览器类 skill

- 如果当前环境提供浏览器类 skill，例如浏览器自动化、页面打开、截图或点击能力，优先在以下场景使用：
  - 页面内容需要展开、翻页、点击后才能看到
  - 官方站点是动态加载页面
  - 论坛、社区、评论区需要真实浏览器查看
  - 需要截图留证或核对页面展示内容

### 3. 专门 skill

- 如果研究过程中遇到明显属于其他专门 skill 的子任务，只调用最小必要集合。
- 例如：
  - 需要导出正式 PDF 时，使用本 skill 自带的 `scripts/md_to_pdf.py`
  - 需要研究 OpenAI 官方产品或 API 时，切到 `openai-docs`
  - 需要操作 GitHub 仓库、PR、issue 时，再调用 GitHub 相关 skill

不要为了“看起来全面”而加载无关 skill。优先保持证据链清晰和上下文干净。

## 搜索策略

至少做两轮以上搜索，不要单次搜索后就下结论。

### 第一轮：搭框架

目标是确认：

- 主要对象是谁
- 主要竞品是谁
- 官方信息入口在哪里
- 用户讨论主要分布在哪些平台
- 市场数据可能来自哪些机构或媒体

推荐搜索词组合：

- `品牌/车型 + 官网`
- `品牌/车型 + 配置`
- `品牌/车型 + 评测`
- `品牌/车型 + 口碑`
- `品牌/车型 + 论坛`
- `品牌/车型 + 销量`
- `细分市场 + 排名`

### 第二轮：补关键证据

围绕第一轮得到的对象和来源，定向补充：

- 发布时间、版本、价格、配置差异
- 真实用户评价和高频痛点
- 竞品对比文章和横评
- 行业统计、研究报告、媒体原创

### 第三轮：查缺补漏

如果以下任一问题还回答不清，就继续搜索：

- 核心竞品是否齐全
- 关键参数是否有官方来源
- 用户痛点是否只有单一来源
- 证据是否过旧
- 不同来源之间是否存在冲突

## 来源优先级

### 产品和配置

优先级从高到低：

1. 官方网站、官方配置表、发布会资料
2. 权威媒体评测、拆解、横评
3. 经整理的汽车数据库
4. 二手转述或无来源参数

### 用户声音

优先级从高到低：

1. 真实车主评论、论坛讨论、访谈记录
2. 社媒公开讨论
3. 媒体总结类内容
4. 营销内容或无来源截图

### 市场信息

优先级从高到低：

1. 官方统计、公司公告、财报、权威研究报告
2. 原创媒体报道
3. 聚合转载
4. 无来源观点

## 输出结构

按来源分组整理，至少包含：

```text
Source
Source Type
URL
Access Date
Published Date
Key Facts
Potential Relevance
Reliability Notes
Conflicts
Evidence Gaps
```

## 四类证据都要收

### Market Evidence

收集：

- 细分市场规模
- 价格带格局
- 主要玩家
- 销量和份额
- 近一年关键变化

### Product Evidence

收集：

- 车型定位
- 版本与配置
- 参数和功能
- 定价与权益
- 媒体评测中的稳定结论

### User Voice Evidence

收集：

- 用户购买原因
- 使用场景
- 高频抱怨
- 高频好评
- 代表性原话摘要

### Competitor Evidence

收集：

- 主要竞品名单
- 产品差异点
- 用户感知差异
- 各自强项和短板

## 记录规则

- 每条关键事实尽量带页面链接或清晰来源名。
- 对用户原话保留原始语义，不要过早翻译成产品需求。
- 对冲突信息并列记录，并写出冲突点。
- 对“看起来像事实但来源可疑”的内容单独标注。
- 对查不到的关键项明确写 `Evidence Gap`。

## 完成检查

完成这一阶段前，确认：

- 是否已经联网，而不是靠记忆补全文字
- 是否至少覆盖四类证据
- 是否优先使用了一手来源
- 是否记录了日期和来源
- 是否保留了冲突和缺口
- 是否还没有越界写产品结论
